关于过滤逻辑对性能的影响思考：
 
- SQL 层面对每个节点做 WHERE use_item_no≠item_no、排除无原料/无子件、排除循环 等判断，会让每次查询都要扫描更多索引或做更多计算。
-如果这些过滤可以提前在 ETL 或数据清洗阶段去掉脏数据，运行期的查询就能更简单、速度更快。
 
-动态过滤（即在每层递归里都做判断）虽保证了结果的准确性，但代价是每次都要重复过滤。
-静态清洗（周期性脚本或数据库视图提前剔除脏数据、环依赖）可以把过滤开销摊平到离线任务，运行期查询就直接对“小而干净”的数据集进行。
